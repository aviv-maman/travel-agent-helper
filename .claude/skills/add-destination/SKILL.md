# add-destination — הוספת יעד מלונות לאתר

## עיקרון-על: דיוק לפני מהירות
הנתונים משמשים לעבודה שוטפת מול לקוחות. **אסור להמציא מרחקים, דירוגים או מתקנים.**
כל מספר מגיע מחישוב (הסקריפט) או ממקור מאומת (קונקטור Booking.com / מחקר רשת
עם סימון ~). נתון חסר → null ומדווחים למשתמש, לא מנחשים.

## סקירת התהליך (6 שלבים)
1. איסוף קלט מהמשתמש ובניית `input.json`
2. הרצת `scripts/build_destination.py` → מרחקים/זמנים מחושבים
3. העשרה מ-Booking.com (קונקטור MCP) → דירוג, קישור, מתקנים
4. יצירת סקשן "על העיר" (info) בסגנון הבית — מחקר רשת, לא מהזיכרון
5. **הצגת אימות למשתמש ולחכות לאישור מפורש** (חובה!)
6. כתיבת `data/destinations/<code>.json` → `bun run seed` → בדיקה באתר → PR

## מבנה היעד בריפו (אל תסטה ממנו)
- הצורה: `SeedDestination` מ-`scripts/extract.ts` (ללא tier — הוסר).
- קובץ היעד: `data/destinations/<code>.json` (code = קוד באותיות קטנות, למשל `bts`).
- רישום: import + שורה במערך `EXTRA_DESTINATIONS` ב-`data/destinations/index.ts`.
- ⚠️ לעולם לא להוסיף יעד ל-`data/seed.json` — `bun run parse` מוחק אותו.
- `bun run seed` בטוח להרצה חוזרת (upsert יעדים, החלפת מלונות ליעד).
- קבצי עבודה זמניים (input/output/geo_cache): `data/destinations/work/<code>/` (ב-gitignore).

## שלב 1 — קלט

לאסוף מהודעת המשתמש:
- שם היעד (עברית + אנגלית) + מדינה + קוד IATA + קוד מדינה ISO (למשל SK)
- רשימת מלונות (שם, ואם ידוע — כוכבים; boards/תגיות רק אם המשתמש ציין)
- נקודות ציון, ולכל אחת: האם היא **רחוב** או **נקודה**

לבנות `data/destinations/work/<code>/input.json` לפי `examples/input_example.json`:
- **שטח (`type: "area"`)**: לרובעים (עיר עתיקה, פלאקה) ולכיכרות-רחבה מוארכות —
  הסקריפט מושך פוליגון מ-Nominatim ומודד עד **גבול השטח** (כמו Google Maps);
  מלון בתוך השטח = 0 מ'. נקודה (`point`) מודדת עד הנעץ עצמו — נכון לכיכרות
  קומפקטיות ולבניינים. אין פוליגון ב-OSM (למשל פלאקה)? נופלים ל-point ומדווחים.
- **רחוב**: `osm_name` חייב להיות השם *המדויק* ב-OpenStreetMap.
  ⚠️ מלכודת ידועה בבודפשט: `Váci utca` (מדרחוב) ≠ `Váci út` (שדרה).
  לא בטוחים? לאמת מול openstreetmap.org / Nominatim לפני ההרצה.
- `start` (north/south/east/west) קובע איזה קצה נחשב "תחילת הרחוב". לשאול את
  המשתמש אם רלוונטי.
- xlsx קיים? לחלץ שמות מלונות בלבד. **לא** להעתיק מרחקים ישנים.

## שלב 2 — הרצת הסקריפט

```bash
cd data/destinations/work/<code>
"C:/Users/danle/travel-agent-helper/travel-agent-backend/.venv/Scripts/python.exe" \
  ../../../../.claude/skills/add-destination/scripts/build_destination.py input.json
```

- stdlib בלבד; ~1–2 דקות ל-20 מלונות (Nominatim מוגבל לבקשה/שנייה).
- פלט: `<city>_output.json` + `<city>_verification.md`; cache ב-`geo_cache.json`.

טיפול בכשלים:
- **מלון לא אותר**: להשיג כתובת (Booking / רשת), להוסיף `address` או `lat`/`lng`, להריץ שוב.
- **רחוב לא נמצא**: לאתר את השם הרשמי ב-OSM ולתקן.
- **בדיקת שפיות**: מרחקים לא הגיוניים (עשרות ק"מ בעיר קומפקטית) → כנראה שני
  רחובות באותו שם; לצמצם bbox או לתת lat/lng ידני.

## שלב 3 — העשרת Booking.com (קונקטור MCP)

לכל מלון ב-output:
1. `accommodations_search` עם שם המלון + העיר → לאתר את הנכס (לוודא התאמת
   שם + קואורדינטות מול מה שכבר חושב).
2. למלא `bookingScore` (דירוג) + `bookingUrl` (קישור לנכס).
3. `answer_property_qa_by_ids` עם שאלות מתקנים: casino? indoor pool?
   outdoor pool? water park? spa? → מיפוי ל-features **רק כשאומת true**:
   `casino` / `pool-in` / `pool-out` / `waterpark` / `spa`. לא ברור → להשמיט.
4. `googleMapsUrl`: `https://www.google.com/maps/search/?api=1&query=<lat>,<lng>`
   מהקואורדינטות שחושבו.

קונקטור לא זמין בסשן → להמשיך עם null, לדווח למשתמש, לא להשלים מהזיכרון.

## שלב 4 — סקשן "על העיר" (info)

לחקור ברשת (WebSearch) ולכתוב he+en **בדיוק בסגנון הקיים** — לפתוח את
`data/seed.json` ולהעתיק את המבנה של BUD/ATH כתבנית:
- `about`: 2–3 משפטי הקשר על העיר מזווית של סוכן נסיעות.
- `attractions`: `"🏛 אתרים מרכזיים: A · B · C · D"` (מפריד ` · `).
- `currencyNote`: `"המטבע המקומי: <שם> (<CUR>) · הערכת שער: ₪1 ≈ <N> <שם>"` —
  שער עדכני מטבלת `exchange_rates` של האפליקציה או מהרשת.
- `airport`: title `"✈ שדה התעופה — <IATA> (<שם>)"` + note מרחק וזמן נסיעה.
- `transport`: 2–4 אפשרויות `{icon: 🚌/🚕/📱/🚆, mode, detail, price: "~6€"}` —
  מחירים אמיתיים עדכניים מהרשת, מסומנים ב-~.
- `warnings`: רק אם באמת רלוונטי (ויזה/ביטוח/אזהרת מטבע).
- `landmarks` (של SeedDestination): key לטיני קצר (`vaci`) + שם Localized לכל
  נקודת ציון — התיאום עם `distances[].landmarkKey` של המלונות.

## שלב 5 — אימות מול המשתמש (חובה, לפני כל כתיבה)

להציג:
1. `<city>_verification.md` — קישורי Google Maps לבדיקה מדגמית (להמליץ על
   2–3 מלונות, כולל אחד "מול אמצע/סוף הרחוב").
2. טבלת מלונות: כוכבים, דירוג Booking, מתקנים שאומתו/חסרים.
3. טיוטת ה-info המלאה (עברית + אנגלית).
4. סיכום: כמה מלונות עובדו/נכשלו.

**לחכות לאישור מפורש.** תיקונים → לעדכן ולהציג שוב.

## שלב 6 — ייבוא ואימות

1. לכתוב `data/destinations/<code>.json` (SeedDestination מלא: code, iata,
   name, country, countryCode, info, sortOrder — הבא בתור אחרי הקיימים,
   landmarks, hotels עם distances/features/stars/bookingScore/urls;
   `rooms: []` תמיד — רשימות חדרים הן DB-managed וממולאות בסקריפט, ראו סעיף 4ב).
2. להוסיף import + שורה ב-`data/destinations/index.ts`.
3. `bun run typecheck && bun run seed`.
4. **העשרת Google Places** (אחרי ה-seed — הנתונים נכתבים ישירות ל-DB):
   `bun scripts/enrich-hotels-places.ts --dest <IATA>` — ממלא דירוג גוגל +
   מספר ביקורות, כתובת, אתר רשמי ותמונה לכל מלון. לעבור על דוח ההתאמות
   בסוף הריצה (שורות ⚠ = התאמה חלשה/חסרה — לתקן ידנית ולהריץ שוב).
   דורש `GOOGLE_PLACES_API_KEY` ב-.env.local. ⚠️ ההעשרה היא DB-managed:
   לעולם לא נכתבת ל-JSON של היעד, ו-seed חוזר משמר אותה לפי שם המלון
   (שינוי שם מלון מנתק את ההעשרה עד ריצה חוזרת של הסקריפט).
   4ב. **רשימות חדרים** (אופציונלי, בתשלום ~$0.01 למלון — לאשר מול המשתמש):
   `bun scripts/enrich-hotels-rooms.ts --dest <IATA> --yes` — מושך מ-Booking
   דרך Apify את חדרי כל מלון (שם, גודל במ"ר, תפוסה) ישירות ל-DB (אותו כלל
   שימור לפי שם מלון). דורש `APIFY_TOKEN` ב-.env.local; בלי `--yes` זו ריצה
   יבשה עם אומדן עלות. לעבור על דוח ההתאמות בסוף.
5. לפתוח `/he/hotels?dest=<IATA>` — לוודא: כל המלונות, RTL, באדג'ים,
   מיון מרחקים במטרים, סקשן "על העיר", ותצוגת ההעשרה (תמונה מימין,
   דירוג גוגל, כתובת מתחת לשם, קישור אתר המלון מתחת ל-Booking).
6. Commit + PR לפי זרימת הבית (branch → PR → המשתמש ממזג).

## כללי עיצוב נתונים (תואם ליעדים הקיימים)
- מרחקים: מטרים (הסקריפט כבר מעגל: ל-10 מ' עד ק"מ, ל-50 מ' מעל).
- זמנים: דקות הליכה; דקות נסיעה **רק כשההליכה ≥ 25 דק'** (הסקריפט כבר
  מדלג על בדיקת הנסיעה מתחת לזה — לא בודקים ולא מציגים).
- הליכה **מעל 40 דק' אינה שמישה** — מציגים נסיעה בלבד (walkMinutes: null).
  הסקריפט כבר מאפס, וגם ה-UI (`hotel-card.tsx`) מסתיר הליכה >40 כשיש נסיעה.
- הסקריפט מסמן ככשל מלון שגיאוקודינג החזיר לו מיקום רחוק מ-50 ק"מ מנקודות
  הציון (עיר אחרת באותו שם — קרה עם "W Prague" → Prague, Nebraska). במקרה
  כזה להשיג קואורדינטות מהקונקטור (accommodations_search מחזיר location)
  ולתת lat/lng בקלט. ⚠️ המטמון שומר גם תוצאה שגויה — למחוק את מפתח
  ה-geo של המלון מ-geo_cache.json לפני ריצה חוזרת.
- בסיס אירוח: `bb` לכל מלון כברירת מחדל; `hb`/`fb` רק אם אומתו — הקונקטור
  בודק זאת עם `meal_plan: half_board` / `full_board` + `hotel_names`
  (מלון שחוזר בתוצאות = מוכר תעריף כזה).
- מחירי תחבורה ב-info: **מטבע יחיד ופורמט קצר** כמו `"~8€"` / `"18–28€"` —
  `toIls()` ממיר כל מספר במחרוזת, ופורמט מעורב ("200 CZK (~8€)") מייצר
  שקלים שבורים. סכום במטבע מקומי → לשדה detail.
- באדג'ים: בריכה מקורה / בריכה חיצונית / קזינו / פארק מים / ספא — רק כשאומתו.
- שמות מלונות: אנגלית (כמו בקיים); שמות יעד/מדינה/נקודות ציון: Localized he+en.
